# Database schemas — EuroTransit

## Orders database

Operator: CloudNativePG  
Cluster CR name: `eurotransit-orders-db` (`postgres/eurotransit-orders-db.yaml`)  
Read/write service: `eurotransit-orders-db-rw.eurotransit.svc:5432`  
Database name: `ordersdb`  
Application user: `app`  
Application secret: `eurotransit-orders-db-app` (generated by CloudNativePG)

Migrations: Flyway in `eurotransit-app` → `backend/orders-service/src/main/resources/db/migration/`

### Order status

`DRAFT` | `RESERVED` | `CONFIRMED` | `FAILED` — stored as **`VARCHAR(50)`**, not a PostgreSQL enum.

`V1__init_orders_schema.sql` created an `order_status` ENUM type, but `V3__fix_status_type.sql`
converted the column to `VARCHAR(50)` and **dropped the type** (R2DBC enum-codec friction — the same
reason Notifications uses `VARCHAR` + `CHECK`). Do not reintroduce `order_status`.

There is **no `PAID` state**: it was residue from the async payment stage and was removed together
with the move to the synchronous authorize (ADR 0018) — see the comment in `Order.kt`.

### `orders`
| Column | Type | Notes |
|--------|------|-------|
| id | UUID PK | server-generated (`UUID.randomUUID()` in `OrderService`) |
| status | VARCHAR(50) | lifecycle state, default `DRAFT` |
| created_at | TIMESTAMPTZ | default `NOW()` |
| updated_at | TIMESTAMPTZ | default `NOW()` |

> `V2__add_order_details.sql` added seven columns (`customer_id`, `route_id`, `seat_class`,
> `quantity`, `total_amount`, `failure_reason`, `version`) for a richer model that was never
> implemented; `V4__drop_unused_order_details.sql` dropped them again. The entity has always been
> the four columns above.

### `processed_events`
| Column | Type | Notes |
|--------|------|-------|
| event_id | VARCHAR(512) PK | dedup key: `{orderId}:{eventType}` (Kafka consumers) |
| processed_at | TIMESTAMPTZ | default `NOW()` |

### `idempotency_records`
| Column | Type | Notes |
|--------|------|-------|
| idempotency_key | VARCHAR(255) PK | HTTP `Idempotency-Key` header / sync API |
| response_payload | JSONB | cached response for duplicate requests |
| created_at | TIMESTAMPTZ | default `NOW()` |

Initial migration: `V1__init_orders_schema.sql` (EM-16).

### `routes`
| Column | Type | Notes |
|--------|------|-------|
| id | UUID PK | route identifier |
| origin | VARCHAR(255) | departure city |
| destination | VARCHAR(255) | arrival city |
| departure_time | TIMESTAMPTZ | scheduled departure |
| total_seats | INT | total capacity |
| available_seats | INT | current available (decremented on reservation) |
| price | DECIMAL(10,2) | ticket price |
| version | INT | optimistic locking counter, default `0` |

### `reservations`
| Column | Type | Notes |
|--------|------|-------|
| id | UUID PK | reservation identifier |
| order_id | UUID | the order this reservation belongs to |
| route_id | UUID FK → routes(id) | the route being reserved |
| seats | INT | number of seats reserved |
| status | VARCHAR(20) | `RESERVED` or `RELEASED` |
| created_at | TIMESTAMPTZ | default `NOW()` |

Unique index: `idx_reservations_order_route ON reservations(order_id, route_id)` — prevents double-reservation at DB level.

## Payments database

Operator: CloudNativePG  
Cluster CR name: `eurotransit-payments-db` (`postgres/eurotransit-payments-db.yaml`)  
Read/write service: `eurotransit-payments-db-rw.eurotransit.svc:5432`  
Database name: `paymentsdb`  
Application user: `app`  
Application secret: `eurotransit-payments-db-app` (generated by CloudNativePG)

Migrations: Flyway in `eurotransit-app` → `backend/payments-service/src/main/resources/db/migration/`

### `payment_intents`
| Column | Type | Notes |
|--------|------|-------|
| id | UUID PK | payment intent identifier |
| order_id | UUID | the order being paid for |
| amount | DECIMAL(10,2) | payment amount |
| currency | VARCHAR(3) | ISO currency code, default `EUR` |
| status | VARCHAR(20) | `AUTHORIZED`, `CAPTURED`, or `FAILED` |
| idempotency_key | VARCHAR(512) | unique key for dedup |
| created_at | TIMESTAMPTZ | default `NOW()` |
| updated_at | TIMESTAMPTZ | default `NOW()` |

Unique indexes:
- `idx_payment_intents_order_id ON payment_intents(order_id)` — prevents double-charge
- `idx_payment_intents_idempotency_key ON payment_intents(idempotency_key)`

### `processed_events` — **unused**
| Column | Type | Notes |
|--------|------|-------|
| event_id | VARCHAR(512) PK | dedup key: `{orderId}:{eventType}` |
| processed_at | TIMESTAMPTZ | default `NOW()` |

> Created by the migration but **never read or written**: Payments stopped consuming Kafka when the
> authorization became synchronous (ADR 0018), so it has no consumer handler to deduplicate.
> Its idempotency is the `idempotency_key` unique index above. Left in place because dropping it
> would need a new migration for no benefit — do not use it as a model for new consumers.

Initial migration: `V1__init_payments_schema.sql` (EM-25).

## Notifications database

Operator: CloudNativePG
Cluster CR name: `eurotransit-notifications-db` (`postgres/eurotransit-notifications-db.yaml`)
Read/write service: `eurotransit-notifications-db-rw.eurotransit.svc:5432`
Database name: `notificationsdb`
Application user: `app`
Application secret: `eurotransit-notifications-db-app` (generated by CloudNativePG)

Migrations: Flyway in `eurotransit-app` → `backend/notifications-service/src/main/resources/db/migration/`

A dedicated database rather than a shared one: the dedup store is what makes "never send twice"
survive a pod restart, so it must not share a failure domain with the money path (ADR 0024,
app ADR-002).

### `sent_notifications`
| Column | Type | Notes |
|--------|------|-------|
| order_id | VARCHAR(255) PK | dedup key — the order id itself, **not** `{orderId}:{eventType}` |
| status | VARCHAR(16) | `PENDING` \| `SENT` \| `FAILED`, default `PENDING`, enforced by a `CHECK` |
| created_at | TIMESTAMPTZ | default `NOW()` |
| updated_at | TIMESTAMPTZ | default `NOW()` |

Two-phase row (`PENDING → SENT`) so that a redelivery after a crash mid-send never sends twice
(app ADR-002/003). `status` is `VARCHAR` + `CHECK` rather than a PG ENUM to avoid R2DBC enum-codec
complexity — the same reason the `orders.status` enum was dropped in V3.

Notifications has **no `processed_events` table**: `sent_notifications` is its dedup store.

Initial migration: `V1__init_notifications_schema.sql`.
