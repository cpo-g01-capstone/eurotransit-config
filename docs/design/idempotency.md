## Why it is mandatory

Kafka delivery is **at-least-once** and remote calls are retried; the network may duplicate
messages. Every handler on the money path must therefore be **idempotent**: applying the same
event/operation twice produces the same result (no double-reserve, no double-charge).

## Idempotency key design

- Primary key: the **`orderId` (UUID)**, generated by Orders at `POST /orders`, propagated in every
  Kafka event (header `idempotency-key`) and in the synchronous call to Payments (`Idempotency-Key`).
- Consumer-side dedup uses a **composite `event_id`** (e.g. `orderId + event-type`) as the primary
  key of `processed_events`, so the same *event* is processed at most once while an order can still
  flow through its different stages.

## Deduplication points (aligned to the current schema)

| Event / operation | Deduplication mechanism | Key |
|-------------------|-------------------------|-----|
| **order-placed** (each consumer) | `processed_events(event_id PK)`, `INSERT ... ON CONFLICT DO NOTHING` before processing, **in the same transaction** as the business change | `event_id` (orderId + event-type) |
| **inventory reserve** | app-level existence check **+** `UNIQUE(order_id, route_id)` on `reservations`; a retried reserve is a no-op | `(order_id, route_id)` |
| **payment authorize** | `UNIQUE(idempotency_key)` on the payments store; if present, return the recorded outcome (no second charge) | `idempotency_key` = `orderId` |
| **order-confirmed** | monotonic state transition (`PAID → CONFIRMED`); confirming an already-`CONFIRMED` order is a no-op | `order_id` |
| **notification-requested** | best-effort; optional dedup | `order_id` |

Dedup happens at **two levels**: the **Kafka consumer** (`processed_events`) *and* the **database**
(UNIQUE constraints / monotonic state). Committing the dedup row in the same transaction as the
business write is what makes redelivery safe.

## Failure scenarios

- **Duplicated `order-placed`** → consumer finds `event_id` in `processed_events` → skip. No double reserve.
- **Retried payment authorize after a timeout, but the auth succeeded** → `UNIQUE(idempotency_key)`
  returns the previous outcome. No double charge.
- **Inventory Pod dies after the reservation commits but before the Kafka offset commit** → on replay,
  the existence check / `UNIQUE(order_id, route_id)` makes the reserve a no-op → "never oversell" holds.
  *(This is chaos experiment CE-2; see `../chaos-experiments/ce-2-pod-kill-inventory-pretest.md`.)*
